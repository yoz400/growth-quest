# 仕様書：週次レビュー可視化強化（草ヒートマップ・目標メーター・前週比）

作成: 2026-07-16 ／ 設計: クロ（Fable 5）／ 実装担当: Codex
対象: `scripts/calendar-review.js`（＋`styles/app.css`少量）
⚠️ 着手条件: 他のJS作業と同時進行しない。
⚠️ **IIFE化後（Phase D完了後）初の機能追加**。新しい関数は calendar-review.js の
**IIFE内に置き、他ファイルから呼ばないので window 公開はしない**（詳細は§5）。

---

## 0. 現状の棚卸し（何が既にあって、何が無いか）

週次レビューは既にかなり可視化されている:
週タブ=トレンド線(6週)・曜日レーダー・ジャンルドーナツ／月・期間タブ=ジャンル積み上げ日別バー／
日タブ=24時間タイムマップ。

**欠けているのは次の3つ**（本仕様のスコープ。1機能=1コミット）:

| # | 機能 | 一言でいうと |
|---|------|------------|
| V-1 | 🌱 継続ヒートマップ（草） | 「半年間つづけてきた軌跡」が一目でわかるGitHub風の草 |
| V-2 | 🎯 目標メーター | 「先週たてた目標」に対する今週の実績バー（今は目標を設定しても翌週ほぼ見えない） |
| V-3 | 📈 前週比の矢印 | サマリーの数字に「先週よりどうだったか」の小さな差分 |

**新しい localStorage キーは無し**（すべて既存データから描画）。台帳追記・export変更も不要。

## 1. V-1: 🌱 継続ヒートマップ（直近26週の草）

### 表示場所と見た目
週タブの「📈 グラフで見る」セクション内、**トレンド線の直下**に追加:

```text
🌱 継続の足あと（直近26週）
  月ラベル:  2月    3月    4月   5月    6月   7月
  月 ░░▓░░█░░░▓▓░░░█░...   ← 7行(月〜日) × 26列(週)
  …
  凡例: なし ░ 少し ▒ ふつう ▓ しっかり █
```

### 実装
`buildStreakHeatmapSVG()` を calendar-review.js の他のSVGビルダー（buildTrendLineSVG等）の
近くに追加。**IIFE内・非公開**。

- データ源: `data.history[YYYY-MM-DD] = 分`（直読み。全期間持っている）
- 範囲: 表示中の週（rvWeekKey）を右端とし、そこから遡って26週分（週の始まりは
  既存の getWeekDates と同じ規則に合わせる）
- 色段階（4段階+なし）: 0分=ほぼ透明 / 1〜14分 / 15〜29分 / 30〜59分 / 60分以上。
  色はアプリのシアン系（`#06b6d4` の透明度違い等）でGQの世界観に合わせる
- セル寸法: viewBox 300×90 目安（セル≈10px+gap1px×26列、7行＋月ラベル行）
- **タップでその日のタイムログへ**: 各セルを
  `<g class="rv-day-bar" data-dk="YYYY-MM-DD">` で包むだけでよい
  （boot.js:554 の委譲が `.rv-day-bar[data-dk]` タップ→ openTimelogModal を既に配線済み。
  月バーと同じ仕組みの流用。**新しいイベント配線を書かない**こと）
- 未来の日付セルは描かない（空白）
- 月の変わり目に上端へ小さな月ラベル（密集回避: 各月の最初の週にだけ）

## 2. V-2: 🎯 目標メーター（先週たてた目標 vs 今週）

### 背景
Section 6「来週の目標」で targetMins を設定できるが、**翌週それがほぼ見えない**
（ジョハリの窓の隠し文言に出るだけ）。設定した目標は堂々と見せて達成感につなげる。

### 表示場所と見た目
週タブの Section 1（今週のサマリー）の**stats grid 直下**:

```text
🎯 先週たてた目標: 120分
   ▓▓▓▓▓▓▓▓░░░░ 85分 / 120分（あと35分）
   達成時 → ▓▓▓▓▓▓▓▓▓▓▓▓ 135分 / 120分 🎉 目標達成！
```

### 実装
- 目標値: `weeklyReviews[前週キー]?.goal?.targetMins`（前週キーは shiftReviewPeriod と
  同じ規則で rvWeekKey から1週戻して得る。既存ヘルパーがあれば流用）
- 実績: 表示中の週の `analyzeWeek(rvWeekKey).totalMins`（renderReviewBody に既にある an を使う）
- **目標未設定（0 or 記録なし）なら、この行ごと非表示**
- 達成時は 🎉＋ゴールド系の色。未達は**責めないトーン**（赤禁止。「あと◯分」の事実だけ）
- 過去週を見ている時も同じロジックで出す（その週の前週の目標 vs その週の実績）

## 3. V-3: 📈 前週比の矢印（サマリー数字の差分）

### 表示場所と見た目
Section 1 の stats grid の各数値の下に小さく:

```text
  8時間20分        12回          5/7
  ▲ +45分         ▲ +3回        ▼ -1日
  総学習時間      セッション数    学習日数
```

### 実装
- renderReviewBody には既に `prev`（前週の analyzeWeek 結果）がある。それを
  Section 1 に渡して差分を計算（totalMins / sessions / studyDays の3つ）
- 表示規則: 増=▲（シアン等の明るい色）／減=▼（**薄いグレー。赤は使わない**＝責めないGQ流）／
  変化なし or 前週データなし=表示しない
- フォントは小さく（.65rem 程度）。数字の主役感を邪魔しない

## 4. 受け入れ基準

V-1 ヒートマップ:
- [ ] 週タブ「グラフで見る」に26週分の草が表示され、学習した日に色が付く
- [ ] 色が学習分数で4段階に変わる（凡例つき）
- [ ] セルをタップするとその日のタイムログが開く（既存の月バーと同じ挙動）
- [ ] 未来の日は空白。データが無い新規ユーザーでも崩れない
- [ ] 週を過去に送る（◀）とヒートマップの範囲も追従する

V-2 目標メーター:
- [ ] 前週に目標を設定していた場合のみ、サマリー直下にバーが出る
- [ ] 達成時に🎉表示。未達時は「あと◯分」（赤色を使っていない）
- [ ] 目標未設定なら行ごと出ない（空バーやゼロ表示を出さない）

V-3 前週比:
- [ ] 3つの数値それぞれに前週比が出る（増=▲明るい色/減=▼薄いグレー）
- [ ] 前週データが無い場合は差分表示なし（NaNや-0が出ない）

共通:
- [ ] 新しい localStorage キーを作っていない
- [ ] 新関数はIIFE内・window公開なし（他ファイルから呼ばれないため）
- [ ] 日/月/期間タブの表示は従来と同一（変更は週タブのみ）
- [ ] スマホ幅375px/320pxでレイアウト崩れなし
- [ ] コンソールエラーゼロ・`bash tools/bump_version.sh` 実行済み

## 5. IIFE化後の作法（Phase D後、初の機能追加なので明記）

- 新関数（buildStreakHeatmapSVG 等）は calendar-review.js の **IIFE の中**に書く
- renderReviewBody からしか呼ばないので **window 公開はしない**（公開は「他ファイルが
  使うものだけ」が Phase D の原則）
- 既存の公開済み関数（openTimelogModal 等）は今までどおり裸で呼べる（window経由で解決）
- もし実装中に「boot.js 側から呼ぶ必要」が生まれたら、§7に従い止まって報告

## 6. テスト手順

1. プレビューで合成データを注入:
   `gq_data` の history に「毎日0〜90分のランダム値×180日分」を入れて reload
2. レビューを開く→週タブ→草が26週分・色段階・月ラベルを確認
3. 草のセルをタップ→その日のタイムログが開く
4. ◀で3週前へ→草の右端が追従している
5. Section 6 で目標120分を設定→保存→翌週分の週へ▶→目標メーターが出る
   （実績を上回る/下回る両方のケースを history 操作で確認）
6. `localStorage.clear()`→reload→新規ユーザーで崩れ・エラーなし
7. 幅320pxで確認

## 7. 迷ったら止まって報告

- 週の始まり規則（getWeekDates）とヒートマップの列がズレる
- 目標データの構造が `goal.targetMins` と違っていた
- rv-day-bar 流用でタップが効かない（委譲の条件に引っかかる等）
- window 公開が必要になった（＝設計と違う依存が生まれた）
