# Growth Quest UI/UX 評価と改善設計（2026-07-03）

クロが「初めて使う新規ユーザー」としてアプリを実際に操作して評価しました。
（オンボーディング9ステップ → ホーム → タイマー → オトモン図鑑 → 設定まわり）

---

## 1. 現状の強み（＝絶対に守るべき資産）

| # | 強み | なぜ価値があるか |
|---|------|------------------|
| 1 | 世界観の一貫性（妖精・召喚の儀・クエスト・冒険者） | 「勉強アプリ」ではなく「冒険」として感情に訴える。愛されるアプリの核 |
| 2 | 「まず5分だけ」の設計 | 行動科学（スモールステップ）として正しい。挫折させない入口 |
| 3 | 段階的解放（ギルド・すごろくは1セッション後に解放） | 初日に全部見せて圧倒しない。「次は何が開くんだろう」というワクワク |
| 4 | オンボーディング後の「導きの妖精」コーチマーク | 最初の行動（タイマーSTART）まで手を引いてくれる |
| 5 | 報酬の多層性（XP・自信・バッジ・オトモン・きずな） | 飽きにくい。複数の「続ける理由」がある |
| 6 | データのエクスポート/復元が設定にある | ユーザーの努力の記録を守る意識がある |

> **コーチマーク** とは：画面の一部を光らせて「ここを押してね」と案内する吹き出しUIのこと。

## 2. 見つかった課題（優先度順）

### 🔴 P0（バグ・信頼を壊すもの。最優先）

**P0-1. オーバーレイ（重ね画面）の多重表示バグ**

実際に確認した証拠：召喚画面（オンボーディング）の表示中に、
裏で「ほめモーダル」「チュートリアル」「妖精ガイド」のボタンも
*同時にクリック可能な状態* になっていました。

```text
いまの状態（悪い例）              あるべき状態
┌─────────────┐        ┌─────────────┐
│ 召喚画面     │        │ 召喚画面     │ ← 表示は常に1つだけ
│ ┌──────────┐│        └─────────────┘
│ │ほめモーダル││ ← 裏で生きている      ↓ 終わったら次へ
│ │┌─────────┐││        ┌─────────────┐
│ ││チュートリアル│││        │ 妖精ガイド   │
│ │└─────────┘││        └─────────────┘
└─┴──────────┴┘
```

- 影響：「スキップ」を押したつもりが別の画面のボタンが反応する。
  タップ位置によっては見えないボタンを押してしまい「壊れてる」印象に。
- 「あとで設定する（スキップ）」が効かない現象もこれが原因の可能性大。

**P0-2. モーダルの排他制御（オーバーレイマネージャ）の導入**

> **排他制御** とは：「同時に1つしか開けない」ように交通整理すること。

設計案（状態機械＝「今どの画面か」を1つの変数で管理する仕組み）：

```text
[閉] ──開く要求──▶ [召喚] ──完了──▶ [妖精ガイド] ──完了──▶ [閉]
                     │
                     └─ 他の画面の「開く要求」は待ち行列に入れ、
                        今の画面が閉じてから順番に開く
```

実装方針：`openOverlay(name)` / `closeOverlay()` を1か所に作り、
既存の各モーダルの表示/非表示を全部この関数経由に置き換える。
（機械的な置き換え作業が多い＝Codex向きのタスク）

### 🟠 P1（UX改善。愛されるアプリへの伸びしろ）

**P1-1. ヘッダーのアイコンナビが「謎アイコン」問題**

現状：🧚📚🥚⚙ が絵文字だけ・ラベルなし。さらにロック中の機能
（🏰ギルド・🎲すごろく）は *空白のすき間* になっていて、壊れて見える。

```text
現状:  [🧚] [📚] (空白) [🥚] (空白) [⚙]   ← 空白が不安を生む

改善案: [🧚] [📚] [❓] [🥚] [❓] [⚙]
        妖精  記録  ???  図鑑  ???  設定
              ↑小さなラベルを常時表示
        ❓＝「Lv◯で解放」とだけ見せて期待感に変える
```

- ロック機能は「隠す」のではなく「シルエット＋？」で見せると、
  段階的解放が *ワクワクの演出* に変わる（ゲームの定石）。

**P1-2. アクセシビリティ（誰でも使えるか）の底上げ**

> **アクセシビリティ** とは：視力・操作環境が違う人でも使えるようにする配慮のこと。

- 絵文字ボタンに `aria-label`（読み上げ用の名前）がない
- 暗い背景に暗いグレー文字の箇所があり、コントラスト（明暗差）不足
- ドラッグ用の「⠿」が何なのか初見では不明 → ツールチップ追加

**P1-3. 冒険者選択の分かりにくさ**

- 「冒険者A/B/C」という名前が無機質（世界観と不釣り合い）
- 選択中の枠線が A と B 両方に見え、どれが選ばれているか曖昧
- 改善：名前をつける（例：レン/ミア/ソラ）＋選択中は1人だけ光る＋
  未選択は少し暗くする

### 🟡 P2（技術的負債。今は動くが、将来の足かせ）

> **技術的負債** とは：急いで作った部分が後で修正コストとして返ってくること。

| 項目 | 現状 | リスク | 対策 |
|------|------|--------|------|
| app.js | 11,281行の1ファイル | 修正時の事故率が上がる | 機能ごとにファイル分割（timer.js / quest.js / otomon.js …） |
| app.css | 4,920行の1ファイル | 同上 | 同様に分割 |
| キャッシュ更新 | `?v=guild-N` を手で上げる | 上げ忘れ→「直ってない」誤認 | ビルド時に自動付与する小スクリプト |
| assets | 166MB。`pet_XXX 2.png` という重複コピーが21個 | リポジトリ肥大・読み込み遅延 | 重複削除＋PNG→WebP圧縮 |

## 4. 分担プラン（消費トークン最適化）

| タスク | 担当 | 理由 |
|--------|------|------|
| 優先順位の判断・設計・仕様の最終決定・コードレビュー | **フェーブル5（クロ）** | 全体を見て「何をやらないか」を決める判断が最重要 |
| オーバーレイ排他制御の実装（機械的な置き換え多数） | **Codex** | 設計が決まれば作業は反復的。トークン単価の安い方へ |
| aria-label付与・コントラスト修正・ラベル追加 | **Codex** | 大量・機械的・仕様が明確 |
| 画像の重複削除・WebP圧縮スクリプト | **Codex** | コマンド作業中心 |
| 冒険者・妖精・ロック中「？」シルエット等のビジュアル | **Canva** | 画像生成・加工はCanvaの領分。コードのトークンを使わない |
| 将来のストア用スクショ・紹介画像・ロゴ | **Canva** | 同上 |
| この設計書・ロードマップ・ユーザーテストのメモ管理 | **Obsidian** | docs/ をObsidianのVault（保管庫）として開けば、リンクで設計書同士をつなげて育てられる |

運用イメージ：

```text
ヨージ ──「これやりたい」──▶ フェーブル5（設計・仕様書づくり・判断）
                                  │ 仕様書（このdocsの形式）
                                  ▼
                            Codex（実装の量産）
                                  │ できたコード
                                  ▼
                            フェーブル5（レビュー・動作確認・マージ判断）

Canva：画像素材を並行して用意 ／ Obsidian：全記録の置き場
```
